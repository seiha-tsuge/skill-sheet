### Q&A投稿サービス

#### 2025/05〜2026/01

#### プロジェクト概要

専門家がユーザーの相談に回答するQ&A投稿型サービスを、自社プロダクトとして企画・開発。
ユーザーが発注前に専門的な知見を獲得できる場を提供することで、知識不足に起因する不適切な発注やトラブルを未然に防止することを目的とした。

本プロジェクトは、

- プロダクトとしての事業価値創出
- 若手エンジニア育成
- 次世代の社内開発標準・基盤の確立

という複数目的を同時に達成する取り組みとして推進。

#### 担当業務

企画から設計・開発までを一貫して主導。スクラムマスターとして4名体制のチームを率い、アジャイル開発を推進しました。

- プロダクトマネジメント
  - 事業価値の定義および要件策定。
  - Figmaを用いたUI/UX設計など、上流工程を包括的に担当。
- 技術・アーキテクチャ設計
  - TypeScriptを用いたフロントエンド／バックエンドの一貫設計。
  - フロントエンド： Feature-Sliced Designを採用し、ディレクトリ構成、状態管理、コンポーネント設計を主導。
  - バックエンド： DDDおよびオニオンアーキテクチャを用い、保守性・拡張性の高いシステム基盤を構築。
- 開発環境・プロセス整備
  - AI活用を前提とした、型安全かつスキーマ駆動の開発基盤およびモノレポ構成の整備。
  - AIと人間が協働しやすいプロセスを確立し、開発生産性を向上。
- 組織貢献・人材育成
  - 本プロジェクトを「次世代の社内開発標準」と位置づけ、設計思想やノウハウを形式知化。
  - コードレビューを通じた若手エンジニアの育成に注力し、技術力と組織力の両面を強化。

### 取り組んだこと

事業仮説としては、
「高い専門性と不確実性を伴う領域では、発注前の相談ニーズが必ず発生する」
という前提を置いていました。

弁護士.comや税理士.comといった既存事例ではこの構造が成立している一方で、
システム開発領域には同様のQ&Aサービスが存在していませんでした。

発注ナビなどの案件を見ても、
要件や予算が固まっていない相談未満の状態が大半であり、
ここに大きな構造的リスクと未充足ニーズがあると考えていました。

そこで「発注」ではなく「相談」にフォーカスし、
納得感が生まれたタイミングで自然に発注につながる導線を
プロダクトとして実装できないか、という仮説で設計しています。

本プロジェクトでは、システム開発における「発注前の相談不足」が生む構造的なリスクに着目し、プロダクト設計を行いました。発注ナビ等に掲載される案件の多くは、要件や予算が固まっていない状態であり、発注者・受注者双方にとって大きな不確実性を抱えています。一方、弁護士や税理士など高い専門性を要する領域では、Q&Aサービスを通じて相談から依頼へとつながるモデルがすでに社会に受け入れられています。そこで本プロジェクトでは、システム開発領域においても「発注」ではなくその手前の「相談」にフォーカスし、専門家とのQ&Aを通じて納得感を醸成した上で、自然に発注・営業管理へとつながる一貫したサービスを目指しました。

プロダクトオーナーとして最も悩んだ意思決定は、相談に回答する「専門家」の信頼性と、サービス立ち上げにおける参加ハードルのバランス設計でした。弁護士や医師などの高度専門職では、資格が専門性と責任の担保として機能しますが、エンジニアは資格がなくても職業として名乗ることができるという構造的な特徴があります。そのため、登録ハードルを下げすぎると回答品質がばらつき、Q&Aサービスとしての信頼性が損なわれるリスクがありました。一方で、登録条件を厳しくしすぎると参加者が集まらず、サービス自体が立ち上がらないという課題も抱えていました。この構造を前提に、本プロジェクトでは個人のスキルではなく「どの組織の一員として回答しているか」を信頼の単位として設計し、個人だけでなく所属法人の明示を必須とすることで、専門性と責任の所在を可視化しました。その結果、品質担保と参加しやすさの両立を図る設計としています。

スクラムを形骸化させないために、スクラムマスターとして最も意識したのは、イベントを省略せず必ず実施することと、プロダクトバックログアイテム（PBI）を戦略的に作成することでした。自身が初めてプロダクトオーナーおよびスクラムマスターを務める立場であったため、独自解釈に陥らないよう、既存プロジェクトで実践されているスクラム運営をベースに進めました。PBI作成においては、GitHub Discussionsを活用し、課題認識や打ち手案を事前に共有・議論することで、チームの思考をバックログに反映する仕組みを構築しました。0→1の開発フェーズにおいても機能の積み上げに終始せず、MVPとして最低限必要な要素やリリースに必要なパスを意識し、「どう打ち出すか」から逆算したPBI設計を行いました。

チーム運営において課題となったのは、開発速度と若手エンジニア育成の両立でした。若手メンバーを中心とした体制では、開発速度を優先するとキャッチアップが追いつかず育成が成立せず、逆に育成に重きを置くとベロシティが大きく低下するというトレードオフが顕在化しました。この状況を踏まえ、本プロジェクトでは両立を無理に目指すのではなく、若手育成を優先する方針に明確に舵を切りました。その結果、開発速度の低下は一定程度受け入れつつ、実装スピードについてはAI活用を前提とした開発体制を構築し、プロダクト推進と人材育成を別の手段で補完する形を取りました。

バックエンドではDDDおよびオニオンアーキテクチャを採用しましたが、特定のドメインが極端に複雑だったわけではありません。0→1開発フェーズであったため、初期段階から完璧なモデリングを目指すのではなく、実装を進めながら違和感が生じたタイミングでリモデリングする方針を取りました。実際にドメインモデルやDBスキーマは何度も破壊と再設計を繰り返し、作っては直すプロセスを通じて理解を深めています。特にDBは変更コストが高い領域ですが、初期フェーズだからこそ許容できると判断しました。DDDを採用した最大の理由は、業務知識を口頭や暗黙知に留めず、コードとして明示的に残すためです。将来的にメンバーの入れ替わりがあっても、仕様や意図がコードから読み取れる状態を維持することを重視しました。

若手にDDDを一度で理解させることは現実的ではないため、値オブジェクトやエンティティといった基本的な概念を実装の中で繰り返し説明し、つまずいたタイミングで都度フォローする形を取りました。DDDを「理解させる」よりも、「業務知識がコードに残る状態を体感させる」ことを重視しています。

フロントエンドではFeature-Sliced Designを採用しました。バックエンドではDDDにより業務知識をコードに残す手法が確立されていますが、フロントエンドにおいても同様に「画面としてドメインをどう表現するか」という知識を構造として残す必要があると考えたためです。FSDのEntityレイヤーを、バックエンドのドメインに対応する「フロントエンドにおけるドメイン表現の層」と位置づけ、モデル定義やAPI利用、表示ルールなどを集約しました。これにより、同一ドメインの表示が画面ごとにばらつくことを防ぎ、知識を一元化しています。一方でFeatureレイヤーは画面遷移や登録処理などの“動詞”に責務を限定し、UI操作や振る舞いに集中させました。結果として、表示（Query）と操作（Command）を分離した構造となり、機能単位の再利用性と保守性を高める設計を実現しました。

Feature-Sliced Designの構造をチームで擦り合わせるにあたり、公式ドキュメントが体系的に整備されている点を重視しました。フロントエンドのディレクトリ設計は属人化しやすい一方で、FSDは責務やレイヤー構造が明確に言語化されており、「このドキュメントを読む」という共通認識を持つことで、設計意図を短時間で共有できました。個別にルールを作り込むのではなく、既存の標準的な設計指針をベースにすることで、チーム内の認識ズレを防ぎ、レビューや実装判断の基準を揃えることができました。

フロントエンドの状態管理設計では、Remixが提唱する Loader → Component → Action のデータフローを軸に設計しました。状態は URL、APIキャッシュ、フォーム、ローカル、グローバルの5種類に整理し、それぞれがどのレイヤーで管理されるべきかを明確にしています。初期データはLoaderで取得し、APIキャッシュはReact Queryで一元管理、コンポーネントでは必要最小限のローカル状態のみを扱う構成としました。特にキャッシュ更新や無効化のロジックが分散しやすい点を課題と捉え、登録や更新といった書き込み操作（Mutation）はFeature単位に集約することで、どの操作がどのキャッシュに影響するかを明示的に管理しています。その結果、状態の責務が整理され、実装・保守ともに見通しの良い構成を実現しました。

AI活用を前提とした開発基盤として、モノレポ構成を採用しました。近年はプロンプトエンジニアリングではなく、AIにどのような文脈を与えるかという「コンテキストエンジニアリング」が重要になると考え、プロジェクト全体のソースコードや設計意図を一貫して読み取れる構成を重視しています。その上で、オニオンアーキテクチャやDDD、CQRSといった設計手法をフロントエンド・バックエンド双方に適用し、人間にとって理解しやすい構造を維持しました。これは「AIが理解しやすいコード」を目指すというより、AIの出力を設計意図に沿ってコントロールできる状態を作るためです。設計方針や前提条件はドキュメントとして明示し、AIエージェントにも共有することで、品質の高いアウトプットを安定して得られる開発プロセスを構築しました。

スキーマ駆動かつ型安全な開発基盤を構築したことで、開発体験と生産性の向上に大きな効果を得ました。モノレポ内にドメインと対になる「contract」パッケージを設け、リクエスト・レスポンスやフォーム入力のスキーマを一元定義し、フロントエンド・バックエンド双方で共通利用しています。フロントエンドではバリデーションや型生成に活用し、バックエンドではRPCやORMと組み合わせることで、DBからUIまでが型で一貫してつながる構成を実現しました。さらにTanStack Startを採用することで、ルーティングを含めたアプリケーション全体の型安全性を担保しています。その結果、変更時の影響範囲がビルド時に即座に可視化され、実行せずとも修正箇所を特定できるようになりました。これにより、安心してコードを変更できる状態が生まれ、開発速度と品質の両立を実現しています。

若手育成を通じて、教え方とマネジメント観が大きく変化しました。従来は概念を論理的に説明すれば伝わると考えていましたが、抽象概念を一方的に解説しても相手の理解や定着につながらない場面が多く、相手側に「参照できる概念」がないと学習は進まないことを痛感しました。そこで認知的意味論の考え方を参考に、複雑な技術概念を日常の比喩に置き換えて説明するアプローチへ転換し、例えばデータベースを「スーパーの陳列（整理されて取り出しやすい状態）」、関数の引数を「炊飯器に入れる米と水」のように、既知の体験に紐づけて理解を促しました。この結果、理解の立ち上がりが早まり、質問の質や自走度の向上につながりました。また、育成やチーム運営では意志の力だけに頼らず、相手への思いやりや感謝を前提にした関わり方が継続的な行動変容につながると学び、コミュニケーションの姿勢を改善しました。

### メール問い合わせ業務システムの新規構築

#### 2025/03〜2026/01

#### プロジェクト概要

メールによる問い合わせ対応業務の非効率を解消するため、既存のメール管理ツール（メールワイズ）に依存した運用を刷新し、問い合わせの自動仕分け・予約情報との紐付け・進捗管理を一体化したオペレーター向け問い合わせ管理アプリを新規開発。

#### 担当業務

### 取り組んだこと

本プロジェクトではフルスタックエンジニアとして参画し、実装を主軸としながら、現場業務の理解から要件整理、PBI作成までを担当しました。最終的な意思決定はPOが行う体制でしたが、顧客・現場課題を整理した上で解決方針を提案し、POの判断を支える役割を担っていました。

本プロジェクトでは、プロダクトオーナーに寄せられた現場からの業務課題を起点に、エンジニアとして業務理解・実装を中心に関与しました。

初期段階では、課題仮説に基づいた簡易実装を行い、それを現場の問い合わせ対応担当者に直接デモ・プレゼンすることで、質疑応答やフィードバックを回収。改善点を反映して再実装する、というサイクルを短期間で繰り返しました。

課題の本質は、施設別・内容別のメール仕分けや分析といった作業が全件人力で行われており、明らかにシステムで代替可能な業務に多くの工数が割かれていた点にあると判断しました。

既存ツールでは自社の予約管理データベースと十分に連携できない制約があったため、新規構築によって問い合わせと予約情報を直接紐づけ、業務全体を一体化する設計を採用しました。結果として、大きな方向性のズレは少なく、主にUI・UX改善を中心としたブラッシュアップを重ねながら、業務効率化を実現しました。

設計上最も悩んだ点は、問い合わせ・旅・人の関係が素直にモデリングするとN対Nに複雑化する点でした。これに対し、問い合わせ単位でスレッドを作成し、スレッドおよびメッセージの両方に必ず旅IDを持たせる設計とすることで、関係性を整理しつつ実運用に耐える構造に落とし込みました。

また、「問い合わせ」「スレッド」「メッセージ」といった概念の認識が人によって異なる点についても、現場とすり合わせを重ねながら共通理解を形成しました。

別PJがメインで、本件はサブプロジェクトとして進めていたため、月1回の頻度で動くものをデモし、質疑とフィードバックを回収して改善するサイクルを回していました。実物があると現場のフィードバックの質が上がるので、ある程度業務理解ができた段階でまず見せることを重視しました。

反復開発なのでスピードは優先しましたが、雑にするのではなく回帰を防ぐためにテストコードを整備して品質を担保しました。フィードバックが頻繁に入るフェーズほど、テストがあると安心して変更できました。

まず業務理解から入り、業務フローを作って共通認識を揃えます。その上でストーリー化してPBIに落とし込み、達成したいことを明確にし、インパクトで優先度をつけて、動くものを見せてフィードバックを早く回します。

従来のPMSでは、客室・食事・アクティビティ・移動などを柔軟に組み合わせたサービス提供が難しいという課題がありました。本プロジェクトではこれらの制約を解消し、顧客一人ひとりに最適化された滞在体験を提供することを目的としました。

現在のオペレーター業務では、TAP社が提供するPMS（宿泊管理システム）を利用している。このPMSは「部屋＋朝食＋夕食」を前提とした固定パッケージ型の予約設計となっており、じゃらんや一休といったOTAとの親和性が高く、業界全体で広く利用されてきた。一方で、アクティビティや移動手段などを柔軟に追加・組み合わせることができず、顧客ごとに最適な体験を設計するうえでは大きな制約となっていた。

そこで目指したのが、客室・食事・アクティビティ・移動手段などを、BTO（パソコンのカスタマイズ）のように自由に組み合わせて予約できる仕組みである。固定パッケージを前提とした従来のPMSから脱却し、より柔軟で拡張性の高い宿泊予約システムを自社で構築することが最終的な目的となった。

ただし、理想形をいきなり実現するのではなく、まずはスモールステップとしてTAP社のシステムから脱却し、自社の基盤となる予約システムを構築することを最初のフェーズとした。この段階での目標は、現行業務を問題なく回せるレベルのシステムを、自分たちの手で作り上げることだった。

開発において最も重要だったのは、業務を正しく理解し、それをシステムとして表現することである。そのため、業務内容を丁寧に洗い出し、ドメイン駆動開発（DDD）の考え方を用いて業務をドメインとして整理・設計した。幸いにも、長年コールセンター業務を経験してきたドメインエキスパート（PO）が存在し、その人物から詳細なヒアリングを行いながら業務知識をドメインに落とし込んでいった。

UI/UXについては、初期段階ではデザインを整えることよりも「まず動くものを作る」ことを優先し、最低限のHTML画面からスタートした。その後、実際のオペレーターに触ってもらいながら高速にフィードバックを回し、PO・現場ユーザー・デザイナーが協力してUXを中心に改善を進めた。業務がスムーズに回る体験を重視し、「どうすれば簡単に予約が取れるのか」を徹底的に検討しながら、段階的にUIを洗練させていった。

その結果、TAPに依存せず、自社の基盤システムだけで基本業務を回せる状態を実現することができた。一方で、最初のフェーズにおける最大の課題は、機能不足やUI以前に、すべての業務を正確に理解し、構造化し、システムとして表現すること自体だった。この段階では、とにかく作り、作ったものを見せ、フィードバックを受けて改善するというサイクルを回し続けることが、そのままユーザーへの価値提供につながっていた。これが「最初の脱TAP段階」である。

その後、1施設での脱TAP成功を踏まえ、複数施設への横展開を進めた。導入自体は順調で、当初は「現行業務に足りない機能を高速に追加する」フェーズが続いたが、やがて運用が安定期に入ると、新たな壁が顕在化した。

実運用が始まると、現場で最も重要視されるのは「安定」である。ヘルプやマニュアル、教育・トレーニング、安定した運用体制が求められる中で、これまでのようにUIや機能を高速で変更し続ける開発スタイルは、毎日同じ業務を回すオペレーターにとって大きな負担となった。「また変わった」「使い方が違う」といったフラストレーションが蓄積し、改善のつもりで行った変更が、かえって現場の不満を生むようになった。

そこで方針を転換し、PO主導でプロダクト側が改善を押し進めるプロダクトアウト型から、現場と一緒に作るマーケットイン型の進め方へと移行した。具体的には、まずデザイナーが現場と直接向き合い、UIや業務フローを一緒に揉む。その上で、PO・スクラムマスター・エンジニアなどプロダクト側で検討・実装を行う流れに変更した。

振り返ると、こちらの「善意の改善」や「新しい価値提供」が、現場にとっては「押し付け」と受け取られていた側面があった。プロダクト側は業務改善や価値向上を目指していたが、現場が強く求めていたのは不満の解消と安定運用であり、その認識のズレが対立を生んでいた。

その中で、特に重要なテーマとして浮かび上がったのが「予約変更の自由化」である。予約後の問い合わせの約4割が、人数・部屋タイプ・プラン・日程などの変更に関するものだった。そこで第一弾の価値提供として、「予約を取り直さずに変更できる」機能に注力した。従来は変更のたびにキャンセルと再予約が必要で、場合によってはキャンセル料が発生していたが、新しい仕組みでは取り直しなしで変更でき、キャンセル料も発生しない方向を目指した。

ただし、この改善はオペレーションを大きく複雑化させ、結果として現場の負担や不満を増やす側面も持っていた。脱TAPを果たした後のフェーズでは、「新しい価値をどこまで提供するか」と「現場の安定運用をどう守るか」のバランスを取ることが、最も難しい課題となった。

複数施設への横展開が進み、運用が本格化していく中で、次に顕在化した課題は「ITヘルプへの問い合わせが大量に発生すること」だった。ここで言うITヘルプとは、オペレーターが日々の業務の中で操作や判断に迷った際に問い合わせる窓口であり、問い合わせが増えるほど現場の対応負荷もプロダクト側の負荷も上がり、運用のボトルネックになっていく。

この問い合わせを減らすことがチームのミッションとなり、現状を丁寧に観察・整理していくと、実際の一次受けは「SV（スーパーバイザー）」に集約されており、SVが解決できないものがITヘルプに上がってくる構造だと分かった。つまり、ITヘルプの問い合わせを減らすには、単に窓口を効率化するのではなく、SVが自力で解決できる状態を増やすことが重要だった。

SVへのヒアリングを進める中で、本質的な原因として浮かび上がったのが「情報が散らばりすぎている」という問題である。SVは状況に応じて、複数のマニュアル、過去の問い合わせ履歴、施設別の手順書、さらには未展開施設については旧システムまで参照しながら対応していた。しかし必要な情報にたどり着くまでの探索コストが大きく、それが結果的に「わからない → 問い合わせ」の増加につながっていた。

現場側でも、ドキュメントを一つのWikiに統合しようとする試みは進められていたものの、既存ドキュメントが膨大で、統合作業は終わりが見えない。そこでプロダクト側は発想を変え、「ドキュメントを全部まとめきる」ことではなく、「今起きている問い合わせをハブにして、必要情報へ到達できる導線を作る」ことを解決策にした。

具体的には、ヘルプ起票（問い合わせ）に対して、解決後に「どうやって解決したか」「どの資料・どの手順を参照したか」を必ず記録してもらう運用へ変更した。これにより、過去の問い合わせを検索し、該当のケースを開けば、解決手順と参照先がまとまって見えるようになる。ヘルプ窓口自体を“情報の集約点（ナレッジハブ）”として設計し、散らばったドキュメントへ到達するための入口にしたのである。この取り組みは、複数のステークホルダーと調整しながら、要件化（PBI化）から実装まで落とし込む必要があり、業務理解と運用設計の両面で大きな挑戦だった。

また、テスト仕様書の作成を任され、試験観点やテストケースの策定に初めて取り組みました。未経験の領域だったため最初は難しさを感じましたが、書籍や記事を通じて自己学習し、不慣れな分野に対する理解を深めました。その結果、単体テストからシナリオテストに至るまでの一連のテスト仕様書を完成させることができました。

#### プロジェクトの採用技術

主要技術はAzure API Management、Azure Functions、Azure Storage、Azure Database for MySQLです。
フロントエンドと実装はTypeScript、Vue.js、Vuetify、Node.js、Sequelizeを使用しました。
