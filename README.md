### メール問い合わせ業務システムの新規構築

#### 2025/03〜2026/01

#### プロジェクト概要

メールによる問い合わせ対応業務の非効率を解消するため、既存のメール管理ツール（メールワイズ）に依存した運用を刷新し、問い合わせの自動仕分け・予約情報との紐付け・進捗管理を一体化したオペレーター向け問い合わせ管理アプリを新規開発。

#### 担当業務

### 取り組んだこと

同時期にもう一つ、新規プロジェクトとして立ち上がったのが「メール問い合わせ業務の刷新」である。従来、問い合わせ対応は「メールワイズ」というメーラー（メール管理ツール）で運用していたが、施設が増えるほど問題が拡大していた。施設ごとにメールアドレスが異なるため、メールを転送して振り分けたり、さらに問い合わせ内容（食事・予約変更など）で手作業で分類したりする必要があり、その“仕分けるだけの作業”が大きな業務コストになっていた。加えて、メールワイズ側では「どの予約（旅）に紐づく問い合わせか」を確認しづらく、メール上で予約変更などの処理もできないため、ツールと業務が分断されていた。

さらに、メールワイズは運用上の制約（施設増加に対してアカウント・アプリ数が不足、コスト増）もあり、拡大に耐えない構造になっていた。結果として、手作業の増加による残業の発生や、対応状況の把握をスプレッドシートに頼らざるを得ない状態が続いていた。

そこでプロダクト側は、メール問い合わせ業務のフローをゼロから理解し直し、従来と同様に「PBIを切る → 動くものを作る → 現場に見せる → フィードバックを得る」を高速で回しながら、自社の問い合わせ管理プラットフォームを構築していった。

新しい仕組みでは、メールを受けた際に「どの旅（予約）に関する問い合わせか」を識別できるキー（予約番号など）を取得し、そこから予約情報を引き当てて画面上で文脈を即座に把握できるようにした。さらに施設別の分類や、問い合わせカテゴリ（食事、変更、キャンセルなど）の振り分けも自動化し、これまで人手で行っていた仕分け作業を大幅に削減した。

また、現場がスプレッドシートで管理していた「担当者の割り当て」「引き継ぎ」「ステータス管理」「残件数」「今日どれだけ処理すれば終われるか」といったオペレーション管理も、プラットフォーム側で完結できるようにした。チェックイン日が近い予約を優先度高として扱えるなど、緊急度・優先度の判断も業務に組み込めるようになり、対応の質とスピードの両方を上げる土台が整った。

このプロジェクトでは、開発だけでなく、資料作成やプレゼンテーション、関係者調整なども積極的に行い、現場に「何を作り、何を変え、どう改善されるか」を継続的に伝えながら推進した。そして、見せてフィードバックをもらい、すぐ改善してまた見せる、という反復を徹底することで、メールワイズからの脱却、スプレッドシートからの脱却、そして問い合わせ業務全体の可視化と効率化を実現し、実際に大きな業務改善へつながった。

現在のオペレーター業務では、TAP社が提供するPMS（宿泊管理システム）を利用している。このPMSは「部屋＋朝食＋夕食」を前提とした固定パッケージ型の予約設計となっており、じゃらんや一休といったOTAとの親和性が高く、業界全体で広く利用されてきた。一方で、アクティビティや移動手段などを柔軟に追加・組み合わせることができず、顧客ごとに最適な体験を設計するうえでは大きな制約となっていた。

そこで目指したのが、客室・食事・アクティビティ・移動手段などを、BTO（パソコンのカスタマイズ）のように自由に組み合わせて予約できる仕組みである。固定パッケージを前提とした従来のPMSから脱却し、より柔軟で拡張性の高い宿泊予約システムを自社で構築することが最終的な目的となった。

ただし、理想形をいきなり実現するのではなく、まずはスモールステップとしてTAP社のシステムから脱却し、自社の基盤となる予約システムを構築することを最初のフェーズとした。この段階での目標は、現行業務を問題なく回せるレベルのシステムを、自分たちの手で作り上げることだった。

開発において最も重要だったのは、業務を正しく理解し、それをシステムとして表現することである。そのため、業務内容を丁寧に洗い出し、ドメイン駆動開発（DDD）の考え方を用いて業務をドメインとして整理・設計した。幸いにも、長年コールセンター業務を経験してきたドメインエキスパート（PO）が存在し、その人物から詳細なヒアリングを行いながら業務知識をドメインに落とし込んでいった。

UI/UXについては、初期段階ではデザインを整えることよりも「まず動くものを作る」ことを優先し、最低限のHTML画面からスタートした。その後、実際のオペレーターに触ってもらいながら高速にフィードバックを回し、PO・現場ユーザー・デザイナーが協力してUXを中心に改善を進めた。業務がスムーズに回る体験を重視し、「どうすれば簡単に予約が取れるのか」を徹底的に検討しながら、段階的にUIを洗練させていった。

その結果、TAPに依存せず、自社の基盤システムだけで基本業務を回せる状態を実現することができた。一方で、最初のフェーズにおける最大の課題は、機能不足やUI以前に、すべての業務を正確に理解し、構造化し、システムとして表現すること自体だった。この段階では、とにかく作り、作ったものを見せ、フィードバックを受けて改善するというサイクルを回し続けることが、そのままユーザーへの価値提供につながっていた。これが「最初の脱TAP段階」である。

その後、1施設での脱TAP成功を踏まえ、複数施設への横展開を進めた。導入自体は順調で、当初は「現行業務に足りない機能を高速に追加する」フェーズが続いたが、やがて運用が安定期に入ると、新たな壁が顕在化した。

実運用が始まると、現場で最も重要視されるのは「安定」である。ヘルプやマニュアル、教育・トレーニング、安定した運用体制が求められる中で、これまでのようにUIや機能を高速で変更し続ける開発スタイルは、毎日同じ業務を回すオペレーターにとって大きな負担となった。「また変わった」「使い方が違う」といったフラストレーションが蓄積し、改善のつもりで行った変更が、かえって現場の不満を生むようになった。

そこで方針を転換し、PO主導でプロダクト側が改善を押し進めるプロダクトアウト型から、現場と一緒に作るマーケットイン型の進め方へと移行した。具体的には、まずデザイナーが現場と直接向き合い、UIや業務フローを一緒に揉む。その上で、PO・スクラムマスター・エンジニアなどプロダクト側で検討・実装を行う流れに変更した。

振り返ると、こちらの「善意の改善」や「新しい価値提供」が、現場にとっては「押し付け」と受け取られていた側面があった。プロダクト側は業務改善や価値向上を目指していたが、現場が強く求めていたのは不満の解消と安定運用であり、その認識のズレが対立を生んでいた。

その中で、特に重要なテーマとして浮かび上がったのが「予約変更の自由化」である。予約後の問い合わせの約4割が、人数・部屋タイプ・プラン・日程などの変更に関するものだった。そこで第一弾の価値提供として、「予約を取り直さずに変更できる」機能に注力した。従来は変更のたびにキャンセルと再予約が必要で、場合によってはキャンセル料が発生していたが、新しい仕組みでは取り直しなしで変更でき、キャンセル料も発生しない方向を目指した。

ただし、この改善はオペレーションを大きく複雑化させ、結果として現場の負担や不満を増やす側面も持っていた。脱TAPを果たした後のフェーズでは、「新しい価値をどこまで提供するか」と「現場の安定運用をどう守るか」のバランスを取ることが、最も難しい課題となった。

複数施設への横展開が進み、運用が本格化していく中で、次に顕在化した課題は「ITヘルプへの問い合わせが大量に発生すること」だった。ここで言うITヘルプとは、オペレーターが日々の業務の中で操作や判断に迷った際に問い合わせる窓口であり、問い合わせが増えるほど現場の対応負荷もプロダクト側の負荷も上がり、運用のボトルネックになっていく。

この問い合わせを減らすことがチームのミッションとなり、現状を丁寧に観察・整理していくと、実際の一次受けは「SV（スーパーバイザー）」に集約されており、SVが解決できないものがITヘルプに上がってくる構造だと分かった。つまり、ITヘルプの問い合わせを減らすには、単に窓口を効率化するのではなく、SVが自力で解決できる状態を増やすことが重要だった。

SVへのヒアリングを進める中で、本質的な原因として浮かび上がったのが「情報が散らばりすぎている」という問題である。SVは状況に応じて、複数のマニュアル、過去の問い合わせ履歴、施設別の手順書、さらには未展開施設については旧システムまで参照しながら対応していた。しかし必要な情報にたどり着くまでの探索コストが大きく、それが結果的に「わからない → 問い合わせ」の増加につながっていた。

現場側でも、ドキュメントを一つのWikiに統合しようとする試みは進められていたものの、既存ドキュメントが膨大で、統合作業は終わりが見えない。そこでプロダクト側は発想を変え、「ドキュメントを全部まとめきる」ことではなく、「今起きている問い合わせをハブにして、必要情報へ到達できる導線を作る」ことを解決策にした。

具体的には、ヘルプ起票（問い合わせ）に対して、解決後に「どうやって解決したか」「どの資料・どの手順を参照したか」を必ず記録してもらう運用へ変更した。これにより、過去の問い合わせを検索し、該当のケースを開けば、解決手順と参照先がまとまって見えるようになる。ヘルプ窓口自体を“情報の集約点（ナレッジハブ）”として設計し、散らばったドキュメントへ到達するための入口にしたのである。この取り組みは、複数のステークホルダーと調整しながら、要件化（PBI化）から実装まで落とし込む必要があり、業務理解と運用設計の両面で大きな挑戦だった。
